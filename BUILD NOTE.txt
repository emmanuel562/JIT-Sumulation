One thing to flag — not a bug, but worth knowing:
In Test 4 the state machine starts at THREAT_CANDIDATE, not IDLE. That's because the test feeds ttc_actual=1.0 and ttc_min=3.0 from step zero — the TTC breach is immediate so it never gets to record an IDLE step. In the real simulation this doesn't happen because the vehicle starts far away and closes gradually. Not a problem.

One genuine issue to watch:
The Kalman closing velocity estimate is slow to warm up. In Test 1 it reads 0.000 m/s for all 6 readings even though the obstacle is clearly closing. This is expected Kalman behaviour — it needs more steps to build confidence in the velocity estimate. In simulate_run() this resolves itself because you have 100 steps. But it means the first few steps of any simulation will underestimate closing velocity slightly.
Not a blocker for Milestone 2. Worth knowing for Milestone 8 polish.


NOTE:
Yes. Clear picture. Here are all the changes made from the original spec:

---

**1. SimulationSession class — entirely new**

The biggest addition. The original spec had no concept of live state persisting across ticks. `simulate_run()` was the only entry point — it ran all 100 steps in one shot and returned everything.

Now `SimulationSession` owns the live state:
- Holds current timestep, elapsed time, rolling history arrays
- Owns the `KalmanTracker`, `JITStateMachine`, and true obstacle distance
- Exposes `step(inputs, dt)` — advances ONE timestep per tick, not all 100
- Exposes `reset(inputs)` and `force_idle()`
- Fixed RNG seed per session so Kalman noise doesn't flicker on every Streamlit rerun

`simulate_run()` is now demoted to a batch testing helper only. Not the primary path.

---

**2. Three UI control modes — entirely new**

Original spec had RUN and RESET only. Now there are three distinct UI states:

| Mode | Meaning |
|---|---|
| IDLE | Stopped, charts cleared, sliders don't advance anything |
| LIVE | Running, ticks advancing automatically |
| STOPPED | Paused mid-run, display frozen, sliders inactive |

And three buttons with precise defined behaviour:

- **RUN** — resets sliders to defaults + resets session + enters LIVE
- **STOP** — pauses, preserves display, enters STOPPED
- **RESET** — halts entirely, forces system to IDLE, clears charts, does NOT reset sliders

---

**3. autorefresh is now conditional**

Original spec didn't specify when autorefresh fires. Now it's explicit: autorefresh runs **only in LIVE mode**. In IDLE and STOPPED it is disabled completely. No wasted reruns.

---

**4. Slider changes during LIVE take effect on next tick**

Original spec was one-shot — sliders set before RUN, fixed for the whole run. Now slider changes during LIVE are picked up on the next tick automatically because Streamlit reruns immediately on slider move and `SimulationSession.step()` reads current input values each tick.

---

**5. Session state keys defined explicitly**

Three keys now documented:
- `ui_mode` — `"idle"`, `"live"`, or `"stopped"`
- `session` — the `SimulationSession` instance
- `history` — accumulated arrays for live chart rendering

---

**6. Demo flow updated**

Original demo was linear — set sliders, hit RUN, watch. New demo is interactive:

- Press RESET → show IDLE state
- Press RUN → simulation starts live with default values
- **Drag sliders while running** — TTC lines respond in real time
- Press STOP to freeze display and talk through it

That last point is the key competition moment — a judge can drag the friction slider down during a live run and watch the system react. That's significantly more impressive than a pre-baked chart.

---

**What didn't change:**

- All six states and all transitions — identical
- TTC formula — identical
- FrictionEstimator — identical
- KalmanTracker — identical
- File separation rule — identical
- All constraints — identical
- Layout — identical

---
